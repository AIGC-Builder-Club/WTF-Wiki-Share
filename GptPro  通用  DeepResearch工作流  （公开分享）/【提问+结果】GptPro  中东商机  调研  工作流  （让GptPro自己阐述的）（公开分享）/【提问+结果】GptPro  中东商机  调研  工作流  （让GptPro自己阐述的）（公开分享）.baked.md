## 1. 项目边界与“可落地”定义

我没有把本项目做成宏观赛道分析，而是把“可落地商机”限定为三类：

**A 类：项目级机会**  
有明确采购方、项目/招标/RFQ/RFI/拍卖名称、编号或截止日期，可进入官方门户购买标书、提交投标、注册供应商或联系采购方。

**B 类：准入级机会**  
没有公开单个项目清单，但官方明确说明：完成供应商注册后可收到招标、进入采购管线、参加 sourcing events 或访问 tender pipeline。这类机会不是“趋势”，而是可执行的渠道建设动作。

**C 类：第三方聚合器捕捉到、但官方页面受登录/JS 限制未能完整读取的机会**  
只在以下条件下保留：有采购方、项目名、截止日期，并且可回到官方平台核验。此类在商机表里单独标注“需官方复核”。

我排除了以下内容：宏观愿景、投资趋势、新闻稿式“将会建设”、没有采购入口的信息、已经明显过期的招标、标题/买方/截止时间缺失的信息、无法验证的社交媒体碎片。

## 2. 初始计划

初始计划分为四步：

第一步，从三国官方采购体系入手，优先找政府公开招标、国企采购、城市/水电/交通/医疗等刚性采购入口。

第二步，按国家建立“主采购门户 + 大型业主门户 + 行业专项门户”矩阵。

第四步，形成两层交付：先完整复盘如何查，再给可跟进商机。

## 3. 计划调整及原因

执行中做了几次调整：

**调整 1：从“大范围商业机会搜索”转为“买方门户反推”。**  
原因是用户明确不接受趋势判断，因此我优先检索买方正在采购什么，而不是讨论哪些行业增长。

**调整 2：沙特政府 Etimad 主清单存在动态加载/登录限制，改为“官方流程 + NUPCO 官方项目 + 第三方聚合器补充”。**  
Etimad 官方说明其平台支持私营部门查看招标、购买 RFP、电子提交投标、评标和授标，但具体招标列表页面不易被文本浏览器完整读取，所以我用官方 Etimad/政府部门页面确认操作路径，再用 NUPCO 官方页面抓医疗集采，用聚合器仅补充需要回 Etimad 核验的项目。([Ministry of Finance Saudi Arabia](https://www.mof.gov.sa/en/tenders/Pages/default.aspx "Tenders and Procurement"))

**调整 3：阿联酋扩大了 DEWA 和 Abu Dhabi Government Procurement Gate 权重。**  
原因是这两个来源给出大量当前项目。DEWA 页面直接列出 open tenders、状态、费用、发布时间、截止时间和在线购买/提交入口；Abu Dhabi Government Procurement Gate 首页显示 116 个 Active Tenders，并暴露了大量最新项目。([DEWA](https://www.dewa.gov.ae/en/supplier/services/list-of-tender-documents "Open Tenders"))

**调整 4：卡塔尔以 Ashghal 为项目级主来源，QatarEnergy/QatarEnergy LNG/KAHRAMAA 作为供应商通道。**  
Ashghal 官方页面直接列出 35 个 open tenders，适合项目级落地；QatarEnergy 普通 tender 列表和 Mushtaryat/KYS 流程更偏准入和登录后机会。([Ashghal](https://www.ashghal.gov.qa/en/Tenders/Pages/ERPTenderdetailes.aspx?Status=Open "e-Tenders"))

**调整 5：对“同日截止”项目保留但降级。**  
2026-06-17 当天仍显示 open 的项目可能在当地时间内尚可操作，也可能已经过时；保留是为了让已有账号/本地伙伴可快速抢投，但不把它们作为新进入者首选。

## 6. 信息筛选标准

我按以下规则筛选：

**必须满足至少一条：**

1. 官方页面显示 open/active 状态；
    
2. 有编号、项目名、截止日；
    
3. 有采购门户、购买标书、提交 bid/RFQ/RFI 或注册供应商入口；
    
4. 官方说明注册后可获取招标或 procurement pipeline；
    
5. 第三方来源有项目名、采购方、截止日，且采购方能回到官方平台核验。
    

**优先级排序：**

- P0：截止日在未来 7 天内、且项目高价值/门槛高/采购方明确；
    
- P1：截止日在 7–30 天内，可准备投标；
    
- P2：长期注册、供应商准入、框架协议、项目管线；
    
- P3：第三方聚合器机会，需官方复核后进入 CRM。
    

**排除标准：**

- 已过期且无延展迹象；
    
- 只有“市场增长”“愿景”“计划投资”；
    
- 只有标题没有买方/截止日/入口；
    
- 社交媒体或广告性内容无法回到官方采购页面；
    
- 官方页面明确显示 no tenders available；
    
- 招标页面为历史年份。

## 7. 遇到的问题与解决方案

**问题 1：中东政府采购门户大量使用动态加载。**  
解决：优先读取官方搜索片段、门户说明页、部门采购页；如果项目列表无法展开，则只保留“注册入口”或用第三方聚合器补充并标注需复核。

**问题 2：语言混合，尤其沙特 NUPCO 阿拉伯标题。**  
解决：保留英文译名 + 原始编号；对医疗类阿文标题只做低解释性翻译，不扩大范围。

**问题 3：同日截止项目仍显示 open。**  
解决：不删除，但标注“极急/可能已错过”，供已有账号/本地伙伴抢投或追踪延期。

**问题 4：有些官方页面只显示门户，不显示项目。**  
解决：把它们归入“准入级机会”，不伪装成项目级商机。

**问题 5：阿布扎比门户显示 116 个活跃项目，但文本环境只展开了首页暴露的最新清单片段。**  
解决：明确说明未穷尽 116 个；仅汇总已展开的高价值条目，并建议复刻时通过 ADGPG 登录/接口完整导出。

## 8. 可复刻操作步骤

另一个 AI 或分析员可以按以下步骤完整复刻：

1. 固定时间基准，例如“2026-06-17 之后仍未截止”。
    
2. 建立字段表：国家、采购方、门户、项目编号、项目名、状态、发布日期、截止日、费用、保证金、联系人、适合供应商、下一步动作、来源可信度。
    
3. 先访问官方总入口：
    
    - 沙特：Etimad、NUPCO、MOJ/MOH/DGA procurement pages、NEOM、RSG、Diriyah；
        
    - UAE：Federal Supplier Register、Dubai eSupply、DEWA、Digital Dubai iSupplier、ADGPG；
        
    - Qatar：政府 e-tendering、Ashghal、QatarEnergy Mushtaryat、QatarEnergy LNG、KAHRAMAA。
        
4. 对每个门户先搜索 “open tenders / active tenders / public tenders / supplier registration / vendor registration / procurement / RFP / RFQ / RFI”。
    
5. 对动态页面，先尝试首页、搜索结果片段、公开 API 端点、分页、排序参数；如果仍无法读取，不提取不存在的信息，只记录为准入入口。
    
6. 对第三方聚合器，只提取“项目名 + 采购方 + 截止日 + 可回官方核验”的条目。
    
7. 删除明显过期、无采购入口、只有市场趋势的内容。
    
8. 将商机分为项目级和入口级，项目级优先。
    
9. 对同一采购方按供应商类型聚类，例如电力水务、ICT/AI/网络安全、医疗耗材、建筑/基础设施、设施管理、教育培训、To C 地块/零售/汽车服务。
    
10. 输出时标注“下一步动作”：注册、购买标书、联系采购、找本地代理、准备保证金、组联合体、进入 CRM。

## 二、其它重要改进

## 6. 把采集系统改成“Fetcher + Parser + Extractor + Scorer”四段式

当前系统主线是公开来源采集到 RawMessage，再 LLM 摘要标签和自动研判。README 里也写得很清楚：公开来源采集 → RawMessage → LLM 摘要/标签/评分/分流 → 事件链/信号簇 → 候选商机 → 研判包 → 周报。 这条线适合雷达，不适合稳定抽取项目字段。

建议重构为：

```text
SourceRegistry
  -> FetcherResult
  -> RawArtifact / RawItem
  -> SourceSpecificParser
  -> StructuredExtractor
  -> OpportunityRecord
  -> ScoringEngine
  -> ActionableOutput
  -> Dossier / WeeklyReport
```

### Fetcher 类型

```python
class FetcherType(StrEnum):
    STATIC_HTTP = "static_http"
    PLAYWRIGHT = "playwright"
    API = "api"
    PDF = "pdf"
    SEARCH_API = "search_api"
    MANUAL_EXPORT = "manual_export"
    EMAIL_ALERT = "email_alert"
```

### Parser 类型

```python
class ParserType(StrEnum):
    RSS = "rss"
    GENERIC_WEB = "generic_web"
    TENDER_TABLE = "tender_table"
    SUPPLIER_PORTAL = "supplier_portal"
    MARKETPLACE_SEARCH = "marketplace_search"
    EVENT_PAGE = "event_page"
    FREE_ZONE_PAGE = "free_zone_page"
    PDF_TABLE = "pdf_table"
```

### 为什么要这样拆

- Fetcher 负责“拿到东西”。
    
- Parser 负责“从这个来源的格式里取条目”。
    
- Extractor 负责“转统一字段”。
    
- Scorer 负责“排序和可落地判断”。
    

这样才能处理官方门户、登录页、PDF、Marketplace、展会页、自由区页，而不是所有东西都塞成 `original_title/original_excerpt`。

## 9. 去重逻辑要从“URL/标题 hash”升级成“机会快照”

现有 `RawMessage` 有 `dedupe_status`，采集时会按 title/url/source 生成 dedupe key。 事件归并也主要靠同一 subject 和 trigger_type。 这对新闻雷达够用，但对 tender 和 marketplace 不够。

建议三层去重：

### 1. 原始内容去重


### 2. 项目机会去重

```text
country + buyer + tender_id
```

没有 tender_id 时：

```text
country + buyer + normalized_title_similarity + deadline_at
```

### 3. 机会快照

不要覆盖旧记录，保留变化：

```python
OpportunitySnapshot:

    seen_at
    title
    deadline_at
    status
    price_or_fee
    source_url
    content_hash
```

- 截止日延期。
    
- open 变 closed。
    
- 项目标题变更。
    
- 标书费用变化。
    
- 同一项目被多个来源提到。

## 10. LLM 不应负责“判断一切”，它应该做结构化抽取和缺口解释

当前 LLM adapter 输出字段只有 summary、fact_tags、judgment_tags、trigger_type、candidate_type、candidate_suggestion、dossier_suggestion、reasoning。 fallback 规则也只识别 security、AI、hiring、partner/integrator 等。

建议把 LLM 任务拆成三个 prompt：

### Prompt A：字段抽取

只允许输出 JSON：

```json
{
  "opportunity_class": "...",
  "buyer_or_channel": "...",
  "title": "...",
  "tender_id": "...",
  "deadline_at": "...",
  "action_url": "...",
  "access_requirements": [],
  "compliance_requirements": [],
  "risk_flags": [],
  "missing_fields": [],
  "evidence": [
    {"field": "deadline_at", "quote": "...", "confidence": 0.91}
  ]
}
```

### Prompt B：机会分类

```text
public_tender / supplier_registration / market_partnership / retail_distribution /
marketplace_sku / event_expo / free_zone / china_outbound_subcontract / weak_signal
```

### Prompt C：行动建议生成

基于已经结构化的字段生成中文摘要、下一步动作、风险说明。

### 强制规则

- 没有证据就填 `null`。
    
- 不允许把宏观新闻升级成商机。
    
- 第三方聚合器机会必须标 `requires_official_verification=true`。
    
- To C SKU 必须给出 `test_channel` 和 `compliance_risk`，否则不能进高优先级。

## 六、最终目标架构

建议目标不是复杂平台，而是这个简单架构：

```text
Source Registry
    ↓
Fetcher Layer
    static / api / playwright / pdf / search / manual export / email alert
    ↓
Raw Artifact Store
    html / json / pdf / screenshot / snippet
    ↓
Parser Layer
    source-specific parsers + generic fallback
    ↓
Structured Extraction
    deterministic rules first, LLM second
    ↓
Opportunity Records
    public tender / supplier portal / market partnership / To C SKU / expo / free zone
    ↓
Scoring + Risk
    source quality + actionability + China fit + compliance + urgency
    ↓
Outputs
    actionable list / weekly report / dossier / workflow replay
```

现有事件链、信号簇、研判包不要删。它们继续服务“早期雷达”。但要新增一条行动型机会线，否则系统会持续偏向“有意思的信号”，而不是“明天就能跟进的商机”。
